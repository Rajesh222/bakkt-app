Overview

User verification is required before they can transact on the platform. Bakkt uses Sumsub for individual users identity verification to ensure regulatory compliance and platform security. This guide shares the KYC flows' details, integration levels, statuses, and API endpoints relevant to individual user verification.

KYC Process Flow

sequenceDiagram
    participant User
    participant App
    participant Bakkt
    participant Sumsub

    App->>Bakkt: POST /user/kyc/applicant
    Bakkt-->>App: Applicant created
    App->>Bakkt: GET /user/kyc/applicant/token
    Bakkt->>Sumsub: Generate token
    Sumsub-->>Bakkt: Token
    Bakkt-->>App: Sumsub token
    App->>User: Show Sumsub Web SDK
    User->>Sumsub: Upload documents
    Sumsub->>Bakkt: Verification complete
    Bakkt->>App: Webhook: KYC status update

Step 1: Create KYC Application

Initialize the KYC process for a user

Endpoint: POST /user/kyc/applicant

const response = await fetch('https://api.bakkt.com/user/kyc/applicant', {
  method: 'POST',
  headers: {
    'Authorization': 'YOUR_API_KEY',
    'unblock-session-id': sessionId,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    address: {
      address_line_1: '123 Main Street',
      post_code: 'SW1A 1AA',
      city: 'London',
      country: 'GB'
    },
    date_of_birth: '1990-05-15',
    source_of_funds: 'SALARY'
  })
});

// Returns 204 No Content - application created

Source of Funds Options:

  • SALARY - Employment income
  • BUSINESS_INCOME - Self-employed/business revenue
  • PENSION - Retirement income
  • OTHER - Other legitimate sources

Step 2: Get Sumsub Token

Get a token to initialize the Sumsub Web SDK for document collection

Endpoint: GET /user/kyc/applicant/token

const response = await fetch('https://api.bakkt.com/user/kyc/applicant/token', {

const data = await response.json();

Use the token to embed Sumsub's identity verification:

<div id="sumsub-websdk-container"></div>

<script>

  const snsWebSdkInstance = idensic.init(

</script>

Alternative: Upload Documents via API

Instead of Sumsub SDK, you can upload documents programmatically.

Endpoint: PUT /user/kyc/document

// Convert image to base64
const base64Image = await imageToBase64(file);

const response = await fetch('https://api.bakkt.com/user/kyc/document', {
  method: 'PUT',
  headers: {
    'Authorization': 'YOUR_API_KEY',
    'unblock-session-id': sessionId,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    content: base64Image,
    document_type: 'PASSPORT',
    document_subtype: 'FRONT_SIDE', // For ID_CARD or DRIVERS
    filename: 'passport.jpg',
    country: 'GB'
  })
});

Document Types:

  • SELFIE - Face photo
  • PASSPORT - Passport (front page only)
  • DRIVERS - Driver's license (requires FRONT_SIDE and BACK_SIDE)
  • ID_CARD - National ID (requires FRONT_SIDE and BACK_SIDE)
  • RESIDENCE_PERMIT - Residence permit
  • UTILITY_BILL - Proof of address
  • OTHER_PROOF_OF_ADDRESS - Bank statement, etc.

Alternative Verification Methods

For users who cannot provide standard ID documents or document-based proof of address, Bakkt supports alternative verification flows via Sumsub.

❗️

Availability of alternative methods depends on region and merchant setup Reach out to Bakkt Support to review the options applicable for your users

Supported alternative verification flows

  • Non-Document ID (non-doc ID)
    Identity verification without uploading an ID document (e.g. liveness and other provider-supported checks). If this flow fails, the user can retry with standard document-based KYC

  • Non-Document Proof of Address (non-doc POA)
    Verification that does not require a document-based proof of address

Regional availability

  • Non-Doc ID: available only for US residents
  • Non-Doc POA: typically used for users outside EUR/GBP where document-based PoA is not required. The exact list is configurable; eligibility is determined by the applicant’s country from the KYC applicant payload

KYC Best Practices

📘

Most common user verification issues & how to avoid them

  • Same document used for ID and Proof of Address (PoA)

    The PoA must be a different document than the one used for identity verification.

  • Screenshot submitted instead of a document

    Screenshots are not accepted.

    IDs: clear photos only no scans allowed

    PoA: clear photos or PDF files

  • PoA document is outdated

    Most of the PoA documents must be issued within the last 6 months.

    Mobile Operator document dated within the past 3 months are accepted.

  • Document image is cropped

    All corners of the document must be fully visible, even if they contain no information.

  • Address Mismatch

    The address on the PoA must exactly match the address provided by the user.

    Please upload a correct document, or update the user verification profile as a merchant.

Rejection Reasons Guide

When KYC fails, users receive specific rejection reasons to guide remediation.

Soft vs Hard Failures

🟠 Soft KYC Failed (SOFT_KYC_FAILED): User can resubmit with corrections

🔴 Hard KYC Failed (HARD_KYC_FAILED): Permanent rejection, no retry possible

Common KYC SOFT Failures

Document Quality

Rejection Label/Cause:

LOW_QUALITY Image too blurry or pixelated

INCOMPLETE_DOCUMENT Part of document cut off

SCAN Document scanned, which is not allowed

DOCUMENT_DAMAGED Document is torn or damaged

UNSATISFACTORY_PHOTOS Overall poor image quality

SCREENSHOTS Screenshot instead of photo

BLACK_AND_WHITE Colour photo required

Solutions:

  • Images should be clear and well-lit
  • All four corners of documents visible
  • Take photo, not screenshot
  • No glare or shadows
  • Text is readable
Document Validity Issues

Rejection Label/Cause:

DOCUMENT EXPIRED

DOCUMENT INVALID Document type or format invalid or not recognised)

NOT_DOCUMENT Wrong type of document uploaded

DOCUMENT_DEPRIVED Document is missing or damaged

DOCUMENT_DAMAGED Document is torn or damaged

DOCUMENT_PAGE_MISSING, MISSING_PAGE Document page is missing

Solutions:

  • Use current, unexpired documents
  • Submit original, unedited photos
  • Upload correct document type
Data Mismatch

Rejection Label/Cause:

DATA_MISMATCH Often a mismatch between the address provided by the user in the text field, and the address from the POA document

DOB_DATA_FORMAT_MISMATCH Common mistakes due to DOB data format

FULL_NAME_ADDRESS_DATA_MISSING_OR_MISMATCH Full name, address data missing or mismatch

DATA_NOT_FOUND_IN_SOURCE Data not found in source

INCONSISTENT_PROFILE Profile data doesn't match documents

WRONG_ADDRESS Address verification failed

REQUESTED_DATA_MISMATCH Data doesn't match request

PROBLEMATIC_APPLICANT_DATA Data quality issues

Solutions:

  • Address matches proof-of-address document word-for-word
  • DOB correct format and matches ID
  • Full name as on ID, same spelling and order
  • Documents are not expire
    • Ensure profile matches documents exactly
    • Photo (not scan/screenshot), colour, full document visible, in focus
  • Verify address with utility bill
  • Double-check all entered information
Selfie Issues

Rejection Label/Cause:

BAD_SELFIE Selfie quality issues

BAD_VIDEO_SELFIE Video selfie problems

BAD_FACE_MATCHING Face doesn't match ID photo

SELFIE_MISMATCH Person in selfie appears different

SELFIE_WITH_PAPER Holding paper in selfie

Solutions:

  • Ensure good lighting for selfie
  • Look directly at camera
  • Selfie shows clear face (no sunglasses, hats or masks)
  • Ensure face matches ID photo

Common KYC HARD Failures

Compliance Issues

Rejection Label/Cause:

PEP Politically Exposed Person

ADVERSE_MEDIA Negative media coverage

SANCTIONS On sanctions list

CRIMINAL Criminal record flags

BLACKLIST / BLOCKLIST On compliance blacklist

REGULATIONS_VIOLATIONS Regulatory violations

Solutions:

  • PEP and ADVERSE_MEDIA are often false positives (e.g. name matches, unrelated media). They are reviewed case by case and may require enhanced due diligence rather than automatic rejection. Contact support for guidance
Document Validity Issues

Rejection Label/Cause:

FORGERY Document tampering detected

GRAPHIC_EDITOR Photo edited/manipulated

Solutions:

  • FORGERY and GRAPHIC_EDITOR are hard rejection reasons (possible fraud). Resubmission is not allowed for these two. Contact support if you require further explanation.

Building Dynamic UI from Requirements

Use the document requirements to build a dynamic UI that adapts to each country:

function getAcceptedDocuments(countryCode, docReqs) {
  const docs = docReqs[countryCode];
  const accepted = [];

  for (const [docType, requirement] of Object.entries(docs)) {
    if (requirement !== 'NOT_SUPPORTED') {
      accepted.push({
        type: docType,
        pages: requirement
      });
    }
  }

  return accepted;
}

// Usage example
const ukDocs = getAcceptedDocuments('GB', kycRequirements);
ukDocs.forEach(doc => {
  console.log(`Accept ${doc.type}: ${doc.pages}`);
});
// Accept passport: FRONT
// Accept id_card: FRONT_AND_BACK
// Accept driving_license: FRONT_AND_BACK

Complete KYC Example

// Complete KYC flow
async function completeKYC(sessionId) {
  // 1. Create KYC application
  await fetch('https://api.bakkt.com/user/kyc/applicant', {
    method: 'POST',
    headers: {
      'Authorization': API_KEY,
      'unblock-session-id': sessionId,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify({
      address: {
        address_line_1: '123 Main St',
        post_code: 'SW1A 1AA',
        city: 'London',
        country: 'GB'
      },
      date_of_birth: '1990-05-15',
      source_of_funds: 'SALARY'
    })
  });

  // 2. Get Sumsub token
  const tokenResponse = await fetch('https://api.bakkt.com/user/kyc/applicant/token', {
    headers: {
      'Authorization': API_KEY,
      'unblock-session-id': sessionId
    }
  });

  const { token } = await tokenResponse.json();

  // 3. Initialize Sumsub SDK
  const snsWebSdk = idensic.init(
    token,
    () => console.log('Documents submitted'),
    (error) => console.error('Error:', error)
  );

  snsWebSdk.launch('#kyc-container');

  // 4. Monitor via webhook or polling
  await monitorKYCStatus(sessionId);
}

async function monitorKYCStatus(sessionId) {
  const interval = setInterval(async () => {
    const user = await bakkt.getUser(sessionId);

    console.log('Current status:', user.status);

    if (user.status === 'FULL_USER') {
      console.log('✅ KYC Approved!');
      clearInterval(interval);
    } else if (user.status === 'HARD_KYC_FAILED') {
      console.log('❌ KYC Failed');
      clearInterval(interval);
    }
  }, 10000); // Check every 10 seconds
}
